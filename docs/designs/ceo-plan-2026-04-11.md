### The diagnosis

The "zero citations" problem the original PRD set out to measure has a
deeper root cause: **AI chat engines with web retrieval cannot cite content
that does not exist in Google's index**. Five of six pages on
ventureoracle.kr are rejected by Google's crawler. The 1 indexed page
surfaces only for a wrong-word-order brand query. The citation tracker as
designed cannot produce a signal until indexing is fixed.

This was flagged in the CEO review's Step 0A Premise Gap #3 ("zero citations
has two causes — unindexable vs unchosen") and by outside-voice finding #10
("bottleneck might be content, not measurement"). The GSC data confirms
both, and adds a sharper third: **the bottleneck is indexing, upstream of
both content and measurement.**

### Sharpened diagnosis (2026-04-11, after Ethan investigated in GSC)

Ethan identified the 4 specific URLs flagged as "Redirect error":
- `/ethan-cho`
- `/concepts/four-lenses-framework`
- `/concepts/optimism-tax`
- `/korea-vc`

**The redirect chain is not broken.** The site lives at `https://www.ventureoracle.kr/`. Non-www requests 301 to www, which is standard behavior.

**The actual issue is a GSC property-mismatch artifact.** The verified GSC property is `https://ventureoracle.kr/` (non-www URL-prefix). The sitemap lists non-www URLs. Google follows the 301 from non-www to www successfully — pages serve fine. But from the non-www property's perspective, the URL "left the property" via redirect, so the property-scoped Pages report labels it as "Redirect error" even though the content is perfectly retrievable and likely already indexed under the www hostname.

**This is much better news than the original reading.** The site is not invisible to AI chat engines. Pages at `www.ventureoracle.kr/*` are almost certainly already in Google's index — the non-www property simply cannot see them. A `site:www.ventureoracle.kr` Google query is the diagnostic to confirm actual indexed-page count.

Ethan has already:
- Requested re-indexing for all 4 URLs
- Clicked "Validate Fix" on the GSC reason rows (validation started 2026-04-11)

## Execution Target (resolves spec-review issue #1 — Railway/Obsidian filesystem)

**Decision:** Run GEO tracker as **local launchd agent on Ethan's Mac**, not on Railway.

**Why:**
1. Obsidian vault is local — Railway-hosted cron cannot write there directly.
   Alternatives (git-backed vault, Obsidian Sync webhook) add a second moving part.
2. Weekly Monday 09:00 KST cadence does not require high-availability infra. Missing
   one Monday per quarter is acceptable.
3. Content_VentureOracle's Railway claim was likely aspirational for the prediction
   engine side; GEO tracker does not need to match that.

**launchd wake-from-sleep caveat (spec review finding):** `StartCalendarInterval` only
fires when the Mac is awake. If the laptop is asleep at Monday 09:00 KST, the job is
deferred to next wake — could be hours or a day late. Two acceptable mitigations,
Eng Review picks:
- **(a)** Relax success criterion from "<= 2 missed runs in 90 days" to **"<= 4 missed
  runs in 90 days"** and live with occasional late runs.
- **(b)** Add a one-time `sudo pmset repeat wakeorpoweron MTWRF 08:55:00` to the
  install script to wake the Mac 5 minutes before the scheduled run.

Default: **(a)**, plus optional (b) as a manual one-liner if Ethan chooses.

**Implication:** "Reuses Railway deploy" from VO is **downgraded** — GEO tracker runs
locally. Content_VentureOracle code is still reused, but the execution path diverges.
SQLite file at `Content_VentureOracle/data/geo.db` is local-file persistence, no
volume mounting needed.

**Open question for Eng Review:** If Content_VentureOracle's prediction engine also
needs Obsidian write access, the same argument may push VO's prediction runs local too,
which would simplify the deploy story further. Flagged for Section 9.

## Accepted Scope (ordered implementation plan)

1. **Package skeleton.** New subpackage `src/geo/` inside Content_VentureOracle with
   `__init__.py`, `engines/`, `evals/`, `schema.py`, `cli.py`.
2. **Engine clients** in `src/geo/engines/`:
   - `claude.py` — thin wrapper over the existing `src/monitor.py` Claude client
   - `openai.py` — `openai` SDK, model `gpt-4o`
   - `perplexity.py` — OpenAI-compatible client,
     `base_url="https://api.perplexity.ai"`, model **`sonar-pro`**. Sources parser
     reads from the **response-root `citations` array**, not the message content
     (Perplexity-specific shape).
   - `gemini.py` — **`google-genai`** SDK (new unified SDK; `google-generativeai` is
     deprecated)
   - All normalize to `EngineResponse(engine, query, text, sources, latency_ms, error)`
     dataclass. Empty sources for engines that don't return any. Raw API response is
     *not* stored in the dataclass — preserved on disk via SQLite `query_results.raw_response` column (see schema fix below).
3. **Queries config.** `config/geo_queries.yaml` — editable by hand, 18 queries across
   3 categories. `framework:` field tags which framework (if any) a query targets,
   used for the feedback loop (see definition below).
4. **Citation classifier.** `src/geo/classifier.py` — single `ask_claude_json()` call
   per response (not two), asking in structured JSON for:
   ```
   {
     "cited": bool,
     "context": "positive" | "neutral" | "negative" | null,
     "position": "early" | "mid" | "late" | null,
     "sources_domains": [str],
     "competitors_mentioned": [
       { "name": str, "adjective_phrase": str, "sector_framing": str,
         "named_as": "institution" | "partner" }
     ]
   }
   ```
   Max output tokens capped at 600.
5. **SQLite schema** at `data/geo.db`:
   ```sql
   CREATE TABLE runs (
     id INTEGER PRIMARY KEY,
     started_at TEXT NOT NULL,  -- ISO 8601
     kind TEXT NOT NULL,         -- 'weekly_citation' | 'monthly_indexability'
     notes TEXT
   );
   CREATE TABLE query_results (
     id INTEGER PRIMARY KEY,
     run_id INTEGER NOT NULL REFERENCES runs(id),
     query_text TEXT NOT NULL,
     query_category TEXT NOT NULL,   -- 'identity' | 'domain' | 'framework'
     framework TEXT,                  -- nullable, e.g. 'founder_intelligence'
     engine TEXT NOT NULL,            -- 'claude' | 'openai' | 'perplexity' | 'gemini'
     response_text TEXT NOT NULL,
     sources_json TEXT NOT NULL,      -- JSON array of {title, url, domain}
     raw_response TEXT,               -- full raw engine JSON, for debugging
     latency_ms INTEGER,
     error TEXT,
     UNIQUE(run_id, query_text, engine)
   );
   CREATE TABLE citations (
     id INTEGER PRIMARY KEY,
     query_result_id INTEGER NOT NULL REFERENCES query_results(id),
     cited INTEGER NOT NULL,          -- 0|1
     context TEXT,                    -- 'positive'|'neutral'|'negative'
     position TEXT,                   -- 'early'|'mid'|'late'
     classifier_version TEXT NOT NULL -- prompt hash
   );
   CREATE TABLE competitor_mentions (
     id INTEGER PRIMARY KEY,
     query_result_id INTEGER NOT NULL REFERENCES query_results(id),
     competitor_name TEXT NOT NULL,
     adjective_phrase TEXT,
     sector_framing TEXT,
     named_as TEXT
   );
   CREATE TABLE indexability_results (
     id INTEGER PRIMARY KEY,
     run_id INTEGER NOT NULL REFERENCES runs(id),
     url TEXT NOT NULL,
     topic TEXT NOT NULL,
     engine TEXT NOT NULL,
     retrievable INTEGER NOT NULL,    -- 0|1
     on_topic INTEGER,                -- 0|1, null if not retrievable
     response_text TEXT,
     UNIQUE(run_id, url, engine)
   );
   ```
   Indexes: `query_results(run_id)`, `citations(query_result_id)`,
   `competitor_mentions(competitor_name)`.

6. **Content feedback loop.**
   - **Definition of "framework":** Any query in `geo_queries.yaml` where `framework:`
     is non-null (e.g., `founder_intelligence`, `edr`, `mau_trap`, `four_lenses`).
   - **Delta math (rate-based, spec review fix):** For each framework F, compute
     `citation_rate_this_week(F) = cited_count / query_count_for_F`. Compare against
     the **14-day rolling mean of the same rate**. If `|delta_rate| >= 0.15` (i.e., a
     change of 15 percentage points), append a dated entry to the Content Backlog
     note. Rate-based avoids false triggers when query counts vary week to week.
   - **Threshold tuning:** Hard-coded at 0.15 for v1. Tune after 4 weeks of data.
   - **First-run guard:** No feedback triggered until at least 2 weeks of data exist.

7. **Classifier golden set.**
   - `src/geo/evals/golden.jsonl` — 25 examples: 10 synthetic (edge cases: no mention,
     ambiguous mention, negative mention, tangential mention, exact framework name

     Seed-phase: synthetic only; after week 1, add real samples.
   - **Regression definition:** Stored expected output JSON per example. Hash-compared
     on re-run. If >= 1 example flips its `cited` or `context` label, eval fails.
     **No CI** — this is a local-run project. Run as a manual gate via
     `python -m geo.evals.run_golden` before any prompt edit is committed.
   - **First-gate composition:** v1 regression gate runs against the 10 synthetic
     examples only; gate expands to 25 after week 1 backfill. Noted explicitly so
     the Success Criterion isn't ambiguous about which 25.

## Success Criteria

- **Operational:** 4 engines queried weekly for 90 consecutive days with **<= 4 missed
  runs** (relaxed from 2 after spec review flagged launchd wake-from-sleep behavior).
  Monthly indexability audit runs 3 times without manual intervention.
- **Partial-failure tolerance:** A weekly run is counted as successful if **at least
  1 of 4 engines** returned a response. Per-engine failures are logged with the
  `error` field populated; other engines proceed. One 30-second backoff retry per
  engine on transient failures. Full retry/failure spec to be finalized in Eng Review.
- **Correctness:** Classifier golden set passes at every prompt edit; one Q2 2026 GEO
  Index draft produced automatically on 2026-06-30 without requiring code changes.
- **Cost:** Monthly API spend stays under $40 ceiling.
- **Operating budget:** Ethan's total hands-on time for this tracker stays under
  **30 minutes/month** (well under the VO "2 hours/month" umbrella principle; leaves
  headroom for VO's prediction tracker to share the budget).
- **Signal:** At least one framework-level citation delta triggers a Content Backlog
  entry within 90 days — proving the feedback loop fires at least once before
  abandonment.

## Strategic Guardrails

- **"< 2 hours / month" operating budget** (VO core principle): every design decision
  must preserve this. Obsidian is the primary reporting surface. Slack is the alert
  layer. **No hosted dashboards, no email for v1.**
- **"No code reading required"** (PRD line 87): `geo_queries.yaml`,
  `venture_oracle_pages.yaml`, `.env` are the three files Ethan touches.
- **Data model anchors to quarterly index from day 1:** every schema decision gets
  asked "does this make Q2 2026 quarterly report easier or harder?"
- **Zero-citation is the 80% case** for the first 3-6 months: every UI decision must
  be designed for the "nothing happened" case first, not the "big spike" case. The
  Slack digest must say something useful when the answer is "still zero, but domain
  X's citations increased 3 → 5".
- **Local execution over hosted:** resolves the Obsidian filesystem dilemma and keeps
  the blast radius on Ethan's own machine.

These scope items were added after the outside-voice pass (Claude adversarial
subagent, Codex fallback unavailable due to auth) flagged blind spots missed by
sections 1-11.

1. **Phase 0 manual pilot** (outside-voice #10): 2-week manual collection before any
   automation build. GO/NO-GO/PIVOT decision gate. See "Phase 0 — Manual Pilot"
   section above.
2. **Demand-seeded query list** (outside-voice #4): half the 18 queries derived
   from Google Search Console / Naver / LinkedIn search auto-complete before the
   pilot starts.
3. **Private-only competitor data** (outside-voice #5): quarterly public report
   talks only about Ethan's citations, frameworks, and trends. Named competitor
   comparisons stay in private SQLite only.
4. **Classifier cross-check** (outside-voice #1): weekly GPT-4o cross-classifies 10%
   of samples (randomly selected). Disagreements logged; if disagreement rate > 15%,
   Slack alert. ~20 LOC addition.
5. **Temperature=0 + N=3 majority vote** (outside-voice #3): every engine call sets
   `temperature=0` where the SDK supports it. For Claude and Perplexity (which drift
   at temp=0), run N=3 calls per query and use majority vote on the citation flag.
   Triples weekly API cost (~$12 → ~$36) but stays within $40 cap. Critical for
   trustworthy time series.
6. **Time-weighted delta math** (outside-voice #7): replace the 14-day rolling mean
   of raw rates with a time-weighted version that divides by days-since-last-run.
   A 14-day gap (2 missed Mondays) is treated as a 14-day window, not 2 runs.
   Prevents spurious Content Backlog triggers on sparse data.
7. **Backlog read-side ritual** (outside-voice #8): weekly Slack digest surfaces the
   latest Content Backlog entries INLINE in the message (not just a file link).
   Ethan's read-side is already the Slack digest; backlog entries become part of
   what gets read. Zero extra ritual needed. Fix to "write-only log" risk.
8. **Celebration state + regression alarm** (outside-voice #9): two new Slack digest
   states — "first citation ever" (louder message, emoji, link to full response) and
   "regression: cited last week, not this week" (distinct warning). Both override
   the flat zero-state format. Preserves the exact moments the project was built
   to detect.
9. **GEO module isolation** (outside-voice #6): `src/geo/` has zero Python-level
   imports from `Content_VentureOracle/src/` (no `from src.monitor import ...`).
   Instead, any needed utility code is COPIED into `src/geo/` (max 3 small helpers
   from VO). Prevents blast radius from GEO bugs reaching VO prediction subsystem.
   Pre-flight checklist item removed since there's no dependency to verify.
10. **Statistical power / significance testing** (outside-voice #2): replace the
    hard 0.15 threshold in the feedback loop with a **one-sided binomial significance
    test** (p < 0.10) against the 14-day baseline rate. Tune after 4 weeks. Avoids
    false triggers on thin data (6 queries × 4 engines = 24 points per framework
    per week — too few for a raw-rate threshold to be meaningful).
